# 관광데이터 공모전 개발 전 설계 및 스택별 로드맵

작성일: 2026-06-24  
대상 서비스: 온길 - 퍼스널 메모리 투어 스케줄러  
기술 스택: Flutter/Dart, Flask, FastAPI, PostgreSQL/PostGIS, Gemma-4-31B-IT

## 1. 서비스 설계 요약

본 서비스는 사용자가 기억하는 과거의 장소, 예를 들어 옛 거주지, 모교, 골목, 추억의 상점을 입력하면 해당 위치를 `추억 좌표`로 변환하고, 사용자가 선택한 이동 방식에 따라 탐색 범위를 다르게 적용해 실행 가능한 여행 스케줄로 자동 구성하는 앱이다. 도보 이동을 선택하면 추억 좌표 기준 3~5km 안의 산책형 관광권을 탐색하고, 차량 이동을 선택하면 3~10km 안의 확장 관광권까지 탐색한다.

공모전 제안서의 핵심 차별점은 일반 관광 앱처럼 유명 장소를 먼저 제안하는 방식이 아니라, 사용자의 기억 속 장소를 여행의 기준점으로 삼고 현재 이동 가능성에 맞춰 탐색권을 조절한다는 점이다. 따라서 개발 설계도 단순한 장소 검색 앱이 아니라 `추억 좌표 -> 이동 방식 선택 -> 도보/차량 탐색 거리 분기 -> 공공 관광 데이터 수집 -> 위치 기반 필터링 -> AI 보조 추천 -> 시간대별 스케줄 생성 -> 방명록/아카이브 축적` 흐름을 중심으로 구성한다.

## 2. 핵심 기능 우선순위

### MVP 필수 기능

1. 추억 장소 입력 및 좌표 변환
   - 주소 입력, 장소명 입력, 키워드 입력을 지원한다.
   - 카카오 로컬 API 또는 지도 API를 통해 위도, 경도를 확보한다.

2. 이동 방식별 관광 인프라 탐색
   - 한국관광공사 TourAPI의 위치 기반 관광정보를 중심 데이터로 사용한다.
   - 관광지, 음식점, 카페, 숙박, 문화시설을 카테고리별로 분류한다.
   - 도보 이동 모드는 추억 좌표 기준 3~5km 범위에서 카페, 공원, 골목길, 소규모 문화시설 같은 도보 친화 장소를 우선 탐색한다.
   - 차량 이동 모드는 추억 좌표 기준 3~10km 범위에서 외곽 명소, 숙박, 중거리 관광지, 대형 문화시설까지 확장 탐색한다.
   - 사용자가 이동 방식을 바꾸면 지도 반경, 후보 장소 수, 예상 이동 시간이 즉시 달라지도록 설계한다.

3. 자동 스케줄 생성
   - 당일치기와 1박 2일 코스를 우선 지원한다.
   - 운영시간, 거리, 카테고리 균형, 이동 동선을 고려해 오전, 점심, 오후, 저녁 순서로 배치한다.

4. 지도 및 리스트 기반 일정 확인
   - Flutter에서 지도 화면과 일정 카드 화면을 함께 제공한다.
   - 사용자는 추천 일정을 저장하거나 일부 장소를 교체할 수 있다.

5. 장소별 방명록
   - 특정 좌표 또는 장소 ID에 사용자 기록을 남긴다.
   - 같은 장소를 검색한 사용자가 이전 방명록을 볼 수 있도록 한다.

### 2차 개발 기능

1. 과거 사진 및 공공 아카이브 연동
   - 문화재청, 국가기록원, 관광 이미지 API를 활용한다.
   - 현재 사진과 과거 사진을 나란히 보여준다.

2. Gemma 기반 개인화 추천 설명
   - 사용자의 여행 성향, 동행 유형, 예산, 이동 선호를 반영한다.
   - 추천 결과에 “왜 이 장소가 포함되었는지” 설명을 생성한다.

3. 사용자 추억 데이터 기반 커뮤니티
   - 같은 동네, 같은 학교, 같은 골목을 기억하는 사용자를 느슨하게 연결한다.
   - 방명록, 사진 업로드, 추억 댓글을 누적한다.

4. 지자체/지역화폐 연계 확장
   - 인구 감소 지역 방문 유도 캠페인과 연결한다.
   - 지역 방문 인증, 지역화폐 리워드, 사진 기증 보상 모델로 발전시킨다.

## 3. 전체 아키텍처 방향

```text

  |
  |-- FastAPI Core Service
        - 장소 검색
        - 관광 데이터 수집
        - 스케줄 생성
        - AI 추천 요청
        - 사용자/방명록 API
        |
        v
PostgreSQL + PostGIS
        - 사용자 프로파일
        - 관광지 메타데이터
        - 일정/코스
        - 방명록
        - 추천 이력
        |
        v
Gemma-4-31B-IT
        - 개인화 추천 보조
```

Flask와 FastAPI를 동시에 사용하는 경우 역할을 분리해야 한다. FastAPI는 비동기 요청, 관광 API 연동, 추천/스케줄 생성처럼 성능과 확장성이 필요한 핵심 서비스에 사용한다. Flask는 관리자용 대시보드, 공모전 시연용 백오피스, 데이터 보정 도구처럼 단순하고 빠르게 구현할 수 있는 영역에 한정한다.

## 4. 스택별 개발 로드맵

### 4.1 Frontend: Flutter / Dart

#### 설계 방향

Flutter 앱은 “추억 장소 입력에서 완성된 일정 확인까지”의 전환을 짧게 만드는 데 집중한다. 첫 화면은 검색창과 지도 중심으로 구성하고, 추천 결과는 지도 경로와 시간대별 카드가 함께 보이도록 설계한다.

#### 주요 화면

1. 온보딩/권한 화면
   - 위치 권한, 서비스 목적 안내, 선택적 로그인 진입을 처리한다.

2. 추억 장소 입력 화면
   - 주소, 장소명, 키워드 입력을 지원한다.
   - 모호한 장소명은 후보 리스트로 보여준다.

3. 이동 방식별 탐색 결과 화면
   - 지도 위에 관광지, 식당, 카페, 숙박 마커를 카테고리별로 표시한다.
   - 도보/차량 이동 선택 세그먼트를 제공한다.
   - 도보 선택 시 3km, 5km 반경을 표시하고, 차량 선택 시 3km, 5km, 10km 반경을 표시한다.
   - 지도에는 현재 선택한 이동 방식의 탐색권을 시각화해 사용자가 도보 친화 관광권과 차량 접근 관광권의 차이를 직관적으로 이해하게 한다.

4. 자동 스케줄 화면
   - 오전, 점심, 오후, 저녁, 숙박 순서로 일정을 표시한다.
   - 장소 교체, 일정 저장, 지도 보기 기능을 제공한다.

5. 장소 상세 화면
   - 현재 관광 정보, 운영시간, 이미지, 과거 사진, 방명록을 보여준다.

6. 내 여행 기록 화면
   - 저장한 코스, 작성한 방명록, 방문한 추억 장소를 관리한다.

#### 상태 관리 전략

초기 개발은 Riverpod 또는 Bloc 중 하나로 통일한다. 공모전 MVP에서는 Riverpod이 적합하다. 검색 조건, 좌표, 추천 결과, 사용자 프로파일, 저장된 일정 상태를 명확히 분리하기 쉽고 코드 양이 비교적 작다.

권장 상태 단위:

- `MemoryPlaceState`: 사용자가 입력한 추억 장소와 좌표 후보
- `NearbyTourState`: 반경 내 관광 인프라 목록
- `ScheduleState`: 생성된 당일/1박2일 일정
- `UserPreferenceState`: 여행 성향, 동행 유형, 예산, 이동 선호
- `GuestbookState`: 장소별 방명록 목록과 작성 상태
- `MobilitySearchState`: 도보/차량 이동 선택값, 적용 탐색 반경, 예상 이동 시간, 결과 부족/과다 상태

#### Flutter 개발 순서

1. 프로젝트 생성 및 라우팅 구조 구성
2. 디자인 토큰, 공통 버튼, 카드, 지도 마커 컴포넌트 작성
3. 추억 장소 입력 화면 구현
4. 검색 결과 지도/리스트 화면 구현
5. 자동 스케줄 카드 화면 구현
6. 장소 상세 및 방명록 화면 구현
7. 저장한 일정/마이페이지 구현
8. API 에러, 로딩, 빈 결과 상태 정리
9. 공모전 시연용 데모 데이터 모드 추가

## 5. Backend 설계: FastAPI / Flask

### 5.1 FastAPI 핵심 서비스

FastAPI는 앱에서 호출하는 대부분의 API를 담당한다. 특히 외부 API 연동, 위치 기반 검색, 스케줄 생성, Gemma 추천 요청은 비동기 처리가 유리하므로 FastAPI에 배치한다.

#### 주요 모듈

- `places`: 좌표 변환, 주변 관광지 조회, 장소 상세 조회
- `schedules`: 자동 일정 생성, 일정 저장, 일정 수정
- `recommendations`: Gemma 추천 요청, 추천 이유 생성
- `users`: 사용자 프로파일, 여행 선호 설정
- `guestbooks`: 장소별 방명록 작성/조회
- `archives`: 과거 사진, 공공 아카이브 조회

#### 핵심 API 예시

```http
POST /api/v1/memory-places/geocode
GET  /api/v1/places/nearby?lat={lat}&lng={lng}&mobility=walk&radius=5000
GET  /api/v1/places/nearby?lat={lat}&lng={lng}&mobility=car&radius=10000
POST /api/v1/schedules/generate
GET  /api/v1/schedules/{schedule_id}
POST /api/v1/schedules/{schedule_id}/save
POST /api/v1/recommendations/explain
GET  /api/v1/places/{place_id}/guestbooks
POST /api/v1/places/{place_id}/guestbooks
GET  /api/v1/places/{place_id}/archives
```

#### 스케줄 생성 로직

1. 추억 좌표, 이동 방식, 반경을 입력받는다.
   - `mobility=walk`: 3~5km 범위만 허용한다.
   - `mobility=car`: 3~10km 범위만 허용한다.
   - 허용 범위를 벗어난 요청은 서버에서 보정하거나 명시적 에러로 반환한다.
2. TourAPI와 보조 로컬 API에서 장소 후보를 수집한다.
3. PostgreSQL/PostGIS에서 거리 기준으로 1차 필터링한다.
   - 도보 모드는 가까운 골목, 카페, 공원, 생활권 문화시설에 가중치를 둔다.
   - 차량 모드는 주차 가능성, 외곽 명소, 숙박, 중거리 이동 가능성에 가중치를 둔다.
4. 운영시간, 휴무일, 카테고리, 평점 또는 연관 관광지 가중치를 반영한다.
5. 카테고리 균형을 맞춘다.
   - 오전: 추억 장소 또는 문화/관광지
   - 점심: 음식점
   - 오후: 관광지/카페/문화시설
   - 저녁: 음식점 또는 야경 명소
   - 1박2일: 숙박 포함
6. 카카오 모빌리티 API 또는 거리 계산 기반으로 이동 방식에 맞는 동선을 정렬한다.
7. Gemma에 추천 설명 생성을 요청한다.

### 5.2 Flask 관리 서비스

Flask는 빠르게 만들 수 있는 내부 도구에 적합하다.

#### 권장 기능

- 외부 API 수집 로그 확인
- 관광지 데이터 누락/오류 수동 보정
- 방명록 신고 관리
- 추천 실패 케이스 확인
- 공모전 발표용 관리자 대시보드

Flask는 핵심 사용자 API와 섞지 말고 `/admin` 또는 별도 포트에서 운영한다. 이렇게 분리하면 FastAPI의 서비스 코드가 관리자 기능 때문에 복잡해지는 것을 막을 수 있다.

## 7. AI: Gemma-4-31B-IT 설계 방향

Gemma는 스케줄 생성의 모든 판단을 대신하기보다, 규칙 기반 스케줄러가 만든 후보를 개인화하고 설명하는 역할로 사용하는 것이 안정적이다. 공모전 심사에서는 데이터 활용성과 실현 가능성이 중요하므로, 핵심 일정 생성은 거리, 운영시간, 카테고리 균형 같은 검증 가능한 로직으로 만들고, Gemma는 감성적 연결과 추천 이유를 강화하는 보조 엔진으로 둔다.

### Gemma 입력 데이터

- 사용자 입력 장소
- 추억 좌표, 이동 방식, 적용 반경
- 사용자 여행 성향
- 동행 유형
- 예산
- 후보 장소 목록
- 운영시간, 카테고리, 거리
- 과거 사진/아카이브 존재 여부
- 방명록 요약

### 프롬프트 설계 원칙

1. 모델이 없는 장소를 지어내지 않게 후보 장소 목록 안에서만 선택하도록 한다.
2. 추천 이유는 감성, 동선, 데이터 근거를 함께 포함하도록 한다.
3. 운영시간, 휴무일, 이동 가능성을 위반하지 않게 규칙을 명시한다.
4. JSON 형태로 응답하게 만들어 Flutter와 FastAPI가 안정적으로 파싱하도록 한다.

### 추천 프롬프트 출력 예시

```json
{
  "theme": "오래된 기억을 현재의 동네 풍경과 연결하는 하루",
  "items": [
    {
      "place_id": "12345",
      "time_slot": "morning",
      "reason": "사용자의 기준 좌표와 가장 가깝고, 과거 사진 비교 경험을 시작하기 좋은 장소입니다."
    }
  ],
  "summary": "추억 장소에서 시작해 가까운 식당과 문화시설을 거쳐 저녁 산책으로 마무리하는 코스입니다."
}
```

## 8. 관광 데이터 정제 파이프라인

1. 외부 API 수집
   - TourAPI 위치 기반 관광정보
   - 국문 관광정보 상세
   - 관광 이미지
   - 카카오 로컬/키워드 검색
   - 문화재청/국가기록원 아카이브

2. 표준화
   - 장소명, 주소, 좌표, 카테고리, 운영시간 필드 정규화
   - 동일 장소 중복 제거
   - 좌표 누락 데이터 보완

3. 품질 검증
   - 이동 방식별 허용 반경 밖 장소 제거
   - 도보 모드는 3~5km, 차량 모드는 3~10km 조건 위반 여부 검증
   - 운영시간 파싱 실패 데이터 표시
   - 이미지 없는 장소 대체 처리

4. 저장
   - PostgreSQL 정형 필드 저장
   - 세부 원문은 `metadata` JSONB에 보존

5. 추천 입력 변환
   - Gemma에 전달할 후보 장소 목록을 짧고 구조화된 JSON으로 변환

## 9. 창의적 서비스 아이디어

### 9.1 추억 좌표 기반 이동 선택형 자동 여행 스케줄러

사용자가 “내가 살던 동네”나 “다니던 학교”를 입력하면 그 장소가 여행의 중심이 된다. 이 추억 좌표를 기준으로 사용자가 도보와 차량 중 이동 방식을 선택하고, 도보는 3~5km, 차량은 3~10km 범위로 TourAPI 관광지, 식당, 카페, 숙박 데이터를 수집한다. PostGIS는 거리와 반경을 계산하고, FastAPI 스케줄러는 이동 방식별 동선을 만든 뒤 Gemma가 감성적 추천 이유를 붙인다.

핵심 조합: 추억 좌표 + 이동 선택 UX + Flutter 모바일 앱 구조 + PostgreSQL/PostGIS + Gemma 추천 설명

### 9.2 과거-현재 비교 타임레이어

장소 상세 화면에서 현재 관광 정보와 과거 사진/기록을 나란히 보여준다. 사용자는 단순히 장소를 방문하는 것이 아니라, 자신의 기억 속 장소가 현재 어떻게 바뀌었는지 확인한다. 공공 아카이브 사진이 없는 경우에는 사용자 기증 사진을 누적해 장기적으로 지역 생활사 데이터가 된다.

핵심 조합: 공공 아카이브 사진 + 관광지 메타데이터 + 사용자 생성 콘텐츠 + 지역 관광 활성화

### 9.3 같은 골목을 기억하는 사람들의 방명록

방명록은 관광지 리뷰가 아니라 특정 좌표에 남기는 기억 기록이다. 같은 학교, 같은 시장, 같은 동네를 찾은 사용자들이 서로의 기록을 볼 수 있다. 이는 단순 리뷰 기능보다 정서적 차별성이 크고, 장기적으로 지역별 추억 아카이브가 된다.

핵심 조합: 사용자 취향 프로파일 + 방명록 + 여행 기대감 + 커뮤니티 소셜 레이어

### 9.4 PostGIS 기반 동선 최적화

추천 장소를 예쁘게 나열하는 것만으로는 실제 여행으로 이어지기 어렵다. PostGIS 거리 계산과 이동시간 API를 활용해 실제 가능한 순서로 일정을 구성해야 한다. 운영시간, 휴무일, 식사 시간, 숙박 여부를 함께 고려하면 “바로 떠날 수 있는 일정”이라는 서비스 가치가 강해진다.

핵심 조합: 여행 코스 그래프 + PostGIS 위치 기반 확장 + MVP 기능 우선순위 + 여행 동선 최적화

### 9.5 지자체 협력형 지역 환류 모델

사용자가 인구 감소 지역의 옛 사진이나 방명록을 남기면 지자체가 이를 지역 아카이브로 활용하고, 방문 인증 또는 사진 기증에 지역화폐 보상을 제공할 수 있다. 서비스는 공공 관광 데이터를 소비하는 데서 끝나지 않고, 사용자 생성 데이터를 다시 공공 데이터 품질 개선에 환류하는 구조로 발전한다.

핵심 조합: 지역 관광 활성화 + 공모전 심사 기준 대응 + 운영 모니터링 대시보드 + UGC 기반 관광 데이터 생태계

## 11. API 명세 작성 기준

모든 API는 Flutter가 안정적으로 사용할 수 있도록 JSON 응답 구조를 고정한다.

공통 응답 예시:

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

에러 응답 예시:

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "NO_NEARBY_PLACES",
    "message": "반경 내 추천 가능한 장소가 부족합니다."
  }
}
```

API 문서는 FastAPI Swagger를 기본으로 사용하되, Flutter 개발자가 보기 쉬운 마크다운 명세도 별도로 유지한다.

## 12. 배포 및 운영 방향

### 개발 환경

- Flutter: Android Emulator, iOS Simulator
- FastAPI: Uvicorn
- Flask: 관리자 서버
- DB: PostgreSQL + PostGIS Docker 컨테이너
- AI: Gemma 모델 서버 또는 외부 추론 엔드포인트

### 운영 환경

- FastAPI는 사용자 API 서버로 운영한다.
- Flask는 관리자 네트워크 또는 인증된 관리자만 접근하도록 제한한다.
- PostgreSQL은 정기 백업과 마이그레이션 관리를 적용한다.
- 외부 API 응답은 캐싱하여 호출 제한과 속도 문제를 줄인다.
- 추천 로그는 개인정보를 제거하거나 최소화해서 저장한다.
